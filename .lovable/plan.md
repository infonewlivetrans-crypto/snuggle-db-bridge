
# План: обучающий первый запуск (onboarding) по ролям

Код не пишем. Ниже — архитектура, экраны, тексты, поэтапный план и риски. В конце — вопросы, чтобы зафиксировать детали до начала работы.

---

## 1. Архитектура решения

### 1.1. Принципы
- **Только фронтенд на старте.** Никаких изменений API, RLS, схемы БД, бизнес-логики, авторизации, маршрутов.
- **Полностью необязательно.** В любой момент: «Пропустить», «Закрыть», «Напомнить позже». Закрытие туром не блокирует UI.
- **Безопасный фоллбэк.** Если что-то в onboarding падает — он молча выключается (ErrorBoundary вокруг провайдера тура), приложение продолжает работать как сейчас.
- **По ролям.** Сценарий выбирается из `useAuth().roles` (уже есть в `src/lib/auth/auth-context.tsx`). Никаких новых ролей не вводим.
- **Не мешает работе.** Подсветка элементов — не модальная (можно кликать вокруг), тур приостанавливается, если целевой элемент не найден.

### 1.2. Слои

```
src/lib/onboarding/
  types.ts                     // Step, Tour, Role, Progress
  tours/
    logist.ts                  // сценарии для логиста/менеджера
    carrier.ts                 // для перевозчика
    driver.ts                  // для водителя
    admin.ts                   // для администратора
  storage.ts                   // чтение/запись прогресса (этап 1: localStorage)
  selectStartTour.ts           // выбор тура по роли + текущему пути
  index.ts

src/components/onboarding/
  OnboardingProvider.tsx       // контекст: текущий тур, шаг, действия
  OnboardingOverlay.tsx        // подсветка + tooltip + кнопки
  OnboardingChecklist.tsx      // мини-чек-лист первых действий
  WelcomeDialog.tsx            // приветственный диалог при первом входе
  RestartOnboardingButton.tsx  // «Пройти обучение заново»
```

Подключение: один раз внутри `AuthGate` (после успешной авторизации), отдельным узлом рядом с `<Outlet />`. Никаких правок существующих страниц на этапе 1 не требуется — селекторы привязываются к уже существующим DOM-элементам по `data-tour="..."` атрибутам, которые добавляем точечно.

### 1.3. Модель шага

```
Step = {
  id: string,
  target: string,          // CSS-селектор, обычно [data-tour="..."]
  title: string,
  body: string,
  placement: 'top'|'bottom'|'left'|'right'|'center',
  action?: 'click'|'navigate'|'none',
  navigateTo?: string,     // если шаг требует перехода на другую страницу
  optional?: boolean,
  waitFor?: () => boolean  // например, появление элемента
}
Tour = { id, role, version, steps: Step[], checklist: ChecklistItem[] }
```

### 1.4. Хранение прогресса
- **Этап 1 (без БД):** `localStorage` ключ `onboarding:v1:{userId}:{tourId}` со значениями `seen|skipped|completed` + `lastStepId`. Привязка к `userId`, чтобы на одном устройстве у разных пользователей не смешивалось.
- **Этап 2 (когда стабилизируем):** колонка `onboarding_state jsonb DEFAULT '{}'` в `public.profiles` + два уже существующих паттерна (RLS «свой профиль»). API не добавляем — пишем через существующий апдейт профиля. Это единственное возможное изменение БД, и оно делается отдельной маленькой миграцией ПОСЛЕ обкатки фронта. На этапе 1 — не трогаем.

### 1.5. Библиотека
- Свой минимальный оверлей (≈150 строк) — без новых зависимостей, чтобы не утяжелять бандл и не конфликтовать с TanStack Start SSR. Альтернатива — `driver.js` (легкий, без React-специфики). Решение зафиксируем после ответа на вопрос ниже.

## 2. Экраны и сценарии по ролям

Общий поток для всех ролей при первом входе:
```
Login → AuthGate → (первый раз для роли?) → WelcomeDialog
   → [Начать тур] → Tour по роли → Checklist остаётся в углу
   → [Пропустить] → Checklist остаётся, тур доступен из меню профиля
```

### 2.1. Логист / менеджер  (роль: logist, manager)
Стартовая страница тура: `/logist` (если включён модуль) либо `/transport-requests`.

Шаги:
1. Приветствие — что такое кабинет логиста.
2. `data-tour="requests-list"` — список заявок: «Здесь появляются заявки на перевозку».
3. `data-tour="create-request"` — кнопка создания заявки.
4. Переход на `/delivery-routes` — «Из заявок собираем маршруты».
5. `data-tour="create-route"` — создать маршрут.
6. `data-tour="assign-carrier"` — назначить перевозчика/водителя.
7. `data-tour="route-status"` — где смотреть статус и геолокацию.
8. Переход на `/route-reports` — «Здесь готовые отчёты по маршруту».
9. Финал + предложение открыть чек-лист.

Чек-лист (первые действия):
- [ ] Создать тестовую заявку
- [ ] Сформировать маршрут
- [ ] Назначить перевозчика
- [ ] Открыть отчёт по завершённому маршруту

### 2.2. Перевозчик / собственник транспорта (carrier)
Стартовая страница: `/carrier-offers`.

Шаги:
1. Приветствие — «Здесь приходят предложения по маршрутам».
2. `data-tour="offers-list"` — входящие предложения, как принять/отклонить.
3. Переход на `/carrier-routes` — активные маршруты.
4. `data-tour="assign-driver"` — назначить водителя на маршрут.
5. Переход на `/vehicles` — «Добавьте свои автомобили, без них нельзя назначить водителя».
6. Переход на `/drivers` — «Добавьте водителей и отправьте им ссылку».
7. `data-tour="driver-invite-link"` — как получить ссылку для водителя.
8. `/carrier-payments` — где смотреть оплаты.
9. Финал + чек-лист.

Чек-лист:
- [ ] Добавить автомобиль
- [ ] Добавить водителя
- [ ] Отправить водителю ссылку
- [ ] Принять первое предложение

### 2.3. Водитель (driver)
Особенность: водитель часто заходит по публичной ссылке `/d/:token` без полноценной сессии. Тур должен работать и в этом режиме (только localStorage по token, без userId).

Стартовая страница: `/driver` или `/d/:token`.

Шаги:
1. Приветствие — «Это ваш маршрут на сегодня».
2. `data-tour="route-points"` — список точек.
3. `data-tour="point-actions"` — как отметить «Доставлено / Не доставлено / Возврат».
4. `data-tour="upload-photo"` — как сделать фото подтверждения.
5. `data-tour="qr-scan"` — где сканировать QR.
6. `data-tour="cash-amount"` — как фиксировать наличные.
7. `data-tour="finish-route"` — завершение маршрута.
8. Финал.

Чек-лист (компактный, всегда виден сверху):
- [ ] Открыть первую точку
- [ ] Загрузить фото
- [ ] Закрыть точку
- [ ] Завершить маршрут

### 2.4. Администратор (admin)
Стартовая страница: `/admin/settings`.

Шаги:
1. Приветствие — что доступно администратору.
2. `data-tour="modules"` — Модули: что включить/выключить.
3. `data-tour="users"` (`/users`) — управление пользователями и ролями.
4. `data-tour="invites"` (`/users/invites`) — пригласить пользователя по ссылке.
5. `data-tour="carriers"` (`/carriers`) — справочник перевозчиков.
6. `data-tour="backups"` (`/backups`) — резервные копии.
7. `data-tour="audit-log"` — журнал действий.
8. `data-tour="system-errors"` — где видеть ошибки.
9. `/first-run` — напомнить, что есть страница первого запуска.
10. Финал.

Чек-лист:
- [ ] Включить нужные модули
- [ ] Пригласить первых пользователей
- [ ] Проверить резервные копии
- [ ] Открыть «Первый запуск» и пройти сквозной тест

## 3. Тексты подсказок (черновик, RU, дружелюбный тон)

Общие кнопки: **Далее**, **Назад**, **Пропустить обучение**, **Завершить**. В конце тура: **Готово** + ссылка «Показать обучение снова».

Образцы:

**Welcome (логист):**
> Добро пожаловать в Радиус Трек! Покажу за минуту, как принять заявку, собрать маршрут и проверить отчёт. В любой момент можно нажать «Пропустить» — обучение всегда доступно из меню профиля.

**Шаг «Создать заявку»:**
> Нажмите «Новая заявка», чтобы добавить груз и адреса. Это можно сделать вручную или импортом из Excel — оба способа равнозначны.

**Шаг «Назначить перевозчика»:**
> Выберите перевозчика из списка. Он получит уведомление и сможет принять или отклонить маршрут.

**Welcome (водитель):**
> Привет! Это ваш маршрут. Покажу, как отмечать точки, загружать фото и завершать маршрут. Если что-то непонятно — позвоните диспетчеру, кнопка справа сверху.

**Шаг «Загрузить фото»:**
> На каждой точке можно приложить фото подтверждения. Если связи нет — фото сохранится и отправится позже автоматически.

**Welcome (перевозчик):**
> Здравствуйте! Покажу, как принять предложение, добавить машину и водителя и отправить водителю ссылку на маршрут.

**Welcome (администратор):**
> Покажу основные настройки: модули, пользователи, приглашения, резервные копии и журнал. Это займёт меньше двух минут.

(Полный список текстов согласуем после утверждения сценариев — ниже в вопросах.)

## 4. План реализации по этапам

Каждый этап — отдельный PR, обратимый, проверяемый отдельно.

**Этап 0. Подготовка (0.5 дня)**
- Согласовать сценарии и тексты по этому документу.
- Зафиксировать список `data-tour` атрибутов и страниц, к которым они привязываются.
- Решение по библиотеке (свой оверлей vs `driver.js`).
- Откат: ничего не делалось.

**Этап 1. Каркас onboarding (только фронт, без подключения)**
- Создать `src/lib/onboarding/*` и `src/components/onboarding/*`.
- `OnboardingProvider` подключить, но **по фиче-флагу** `VITE_ONBOARDING_ENABLED=false`. По умолчанию выключено.
- Добавить страницу `/admin/onboarding-preview` (только для роли admin) — внутренний предпросмотр сценариев, в проде скрыт.
- Откат: удалить папки, фича-флаг убрать. На UX не влияет.

**Этап 2. Расстановка `data-tour` атрибутов**
- В существующие страницы добавить только `data-tour="..."` атрибуты — без изменения логики, классов, обработчиков.
- Делаем отдельным маленьким PR по каждой роли (4 PR).
- Откат: убрать атрибуты — UI не меняется.

**Этап 3. Включение welcome + первого тура для одной роли (admin)**
- Включить флаг для admin, обкатать на одном пользователе.
- Прогресс — `localStorage`. БД не трогаем.
- Откат: выключить флаг.

**Этап 4. Включение для остальных ролей по очереди**
- carrier → logist/manager → driver. По одной роли за раз.
- Между этапами — пауза 1–2 дня для обратной связи.

**Этап 5. Чек-лист первых действий**
- Компактный виджет в углу, считает выполнение по фактическим данным (запросы, которые уже есть в `/first-run`: есть ли маршрут, есть ли фото и т.п.). Никаких новых API.
- Откат: скрыть виджет фича-флагом.

**Этап 6. Кнопка «Пройти обучение заново»**
- В меню профиля и на `/first-run`. Сбрасывает localStorage по нужному туру.
- Откат: убрать пункт меню.

**Этап 7. (Опционально, только при необходимости) Перенос прогресса в БД**
- Отдельная маленькая миграция: `profiles.onboarding_state jsonb default '{}'`.
- Чтение/запись через **существующий** апдейт профиля. Новых эндпоинтов и новых RLS не добавляем.
- Делается ТОЛЬКО после стабильной работы этапов 1–6 минимум 2 недели.
- Откат: миграция-«down» удаляет колонку (данные тура не критичны).

## 5. Риски и как их избежать

| Риск | Митигация |
|---|---|
| Сломать существующий UI селекторами/оверлеем | Свой оверлей рендерится через portal, position: fixed; z-index изолирован; тур обёрнут в ErrorBoundary, при ошибке молча выключается. |
| Тур ломается, если элемент не найден (изменили вёрстку) | `waitFor` с таймаутом 3 сек → если не найден, шаг помечается «пропущен», тур продолжается. Без алертов пользователю. |
| Конфликт с модальными окнами/диалогами | Тур не блокирует клики; при открытии модалки шаг автоматически приостанавливается. |
| SSR/гидрация (TanStack Start) | Provider и оверлей рендерятся только на клиенте (`useEffect` + проверка `typeof window`). |
| Водитель по публичной ссылке без userId | Хранить прогресс по `token` (уже есть в URL), отдельный namespace в localStorage. |
| Накопление localStorage между релизами | Версионирование ключа: `onboarding:v{N}:...`. При повышении версии старые ключи игнорируются. |
| Нагрузка на БД | На этапах 1–6 в БД ничего не пишем. Этап 7 — одна колонка jsonb, апдейт через существующий путь. |
| Регрессии в авторизации | Onboarding подключается строго ВНУТРИ `AuthGate`, после всех существующих проверок. До показа `LoginPage` ничего не делает. |
| Производительность бандла | Свой оверлей или динамический импорт `driver.js` через `lazy()`. Загружается только после логина. |
| Локализация и тон текста | Все строки в одном файле `tours/*.ts`, легко править без перекомпиляции логики. |

## 6. Что строго НЕ трогаем
- API роуты и их форматы
- Авторизацию, `AuthGate`, ролевую модель, RLS
- Импорт Excel, создание перевозчиков/машин/заявок
- Существующие страницы (кроме добавления `data-tour` атрибутов)
- `supabase/config.toml`, `.env`, авто-генерируемые файлы Supabase
- Маршруты TanStack (новые маршруты только: `/admin/onboarding-preview` за фичефлагом и опционально пункт в меню профиля)

## 7. Уточняющие вопросы перед реализацией

1. **Библиотека:** свой минимальный оверлей (≈+5 КБ, полный контроль) или `driver.js` (≈+10 КБ, готовый)? По умолчанию предлагаю свой.
2. **Хранение прогресса на этапе 1:** подтверждаете `localStorage` (без БД)? Перенос в `profiles.onboarding_state` — отдельным шагом позже, по необходимости.
3. **Запуск тура:** только при первом входе после регистрации, или также после обновления версии тура (например, добавили новый шаг)?
4. **Роли:** ваши роли в `roles.ts` — `admin`, `logist`, `manager`, `carrier`, `driver`. Если у пользователя несколько ролей — какой тур показывать первым? Предлагаю по приоритету: admin → logist → manager → carrier → driver.
5. **Водитель по публичной ссылке `/d/:token`:** показывать ему мини-тур (4 шага) или достаточно одной всплывающей подсказки на первом экране?
6. **Чек-лист:** показывать всем ролям или только логисту/перевозчику/админу (для водителя избыточно)?
7. **Точка входа «Пройти заново»:** меню профиля, страница `/first-run`, или оба места?
8. **Тон текстов:** «Вы» (как сейчас в черновике) или «ты»?
9. **Должен ли admin видеть туры других ролей** для проверки (через `/admin/onboarding-preview`)?
10. **Включение в production:** через фича-флаг в коде или через настройку в `system_settings` (тогда админ может включать/выключать без релиза — но это потребует чтения существующего эндпоинта `/api/system-settings`, БД не меняем)?

После ваших ответов зафиксирую финальный список шагов, тексты и приступим к Этапу 0–1.
